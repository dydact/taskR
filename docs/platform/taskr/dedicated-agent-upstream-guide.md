# Dedicated Agent Upstream Handoff

This note gives Exo, DeptX, and Flow the minimal contract to start emitting TaskR dedicated-agent reservations/events. Pair it with the fixtures under `docs/fixtures/dedicated/` and the Postman collection in `docs/postman/taskr-dedicated-agents.postman_collection.json`.

---

## 1. Endpoints & Headers

| Service action | HTTP call | Required headers |
| --- | --- | --- |
| Create/update assignment | `POST /dedicated/assignments` | `Authorization`, `X-Tenant-Id` |
| Append timeline event | `POST /dedicated/events` | same |
| Live roster stream (UI) | `GET /dedicated/assignments/stream?tenant=<slug>` | `Authorization` (optional for UI when proxied) |

- Feature flag: `dedicated.agents` must be enabled for the tenant. Staging already has the override; production tenants will be toggled during pilot.
- Payload schema: shared via `common_agents.assignment`. Import directly rather than re-declaring fields.
- Identity: the path honours tenant slug or UUID. We standardised on slugs (`demo`, `acme-industries`) for pilot.

---

## 2. Required Fields

| Field | Notes |
| --- | --- |
| `assignment_id` | Stable UUID generated by the emitter (Exo reservation ID, DeptX workflow session, etc.). |
| `tenant_id` | Matches the slug seen by TaskR. |
| `agent_slug` | Human-readable handle (`xoxo.plan-runner`). Drives roster grouping. |
| `status` | Accepts `reserved`, `active`, `paused`, `released`, `completed`, `failed`, `cancelled`. `failed`/`cancelled` trigger notifications. |
| `priority` | `low`, `normal`, `high`, `critical`. |
| `overlay` | Provide `overlay_id` + `overlay_type`; include persona metadata if available. |
| `capabilities` | CPU/memory/GPU/TEE snapshot. Used for future scheduling UI. |
| `model` | Family/size/provider. |
| `prompt_profile` or `prompt_history` | Optional but keeps the detail drawer meaningful. |
| `polaris_obligations` | Required when Polaris has allocated pools/zones. |
| `metadata` / `context` | Free-form dictionaries. Include links back to Exo/DeptX/Flow control planes. |

### Events

Key types we expect:

| `event_type` | Recommended payload fields |
| --- | --- |
| `assignment.status_changed` | `status`, `node_id` (if attached), reason codes. |
| `assignment.node_attached` / `assignment.node_detached` | `node_id`, `reason` (for detach). |
| `assignment.prompt_appended` | `role`, `content`, `created_at`. |
| `assignment.capabilities_updated` | Same shape as `capabilities`. |

Sending a status change to `failed` or `cancelled` will drive Slack notifications. Include `reason` in the payload so the notification copy is actionable.

## 4. Latency & Retry Expectations

- Endpoints run behind FastAPI; latency <50 ms inside the cluster.
- Retries: recommended exponential backoff capped at 5 attempts. `409` is returned if the assignment/event already exists.
- Order: send a reservation first, then events. We upsert on each call, so idempotency is handled server-side.

## 5. Verification Checklist

1. `curl` + Postman: exercise `/dedicated/assignments` and `/dedicated/events` using the sample payloads.
2. Tail the SSE stream: `curl -H "Accept: text/event-stream" "$BASE/dedicated/assignments/stream?tenant=demo"`.
3. Load TaskR UI → switch to the `xOxO` (dedicated) view → confirm roster entry, detail drawer, timeline updates in real time.
4. Trigger a `failed` status event → verify Slack notification (pilot workspace channel: `#taskr-dedicated-pilot`).
5. Optional: run `python scripts/seed_dedicated.py --tenant demo --token <token>` to reseed demo data during local troubleshooting.
